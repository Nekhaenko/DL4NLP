## Фреймворк ответа (запомни как скелет)

### 1. Functional requirements (5 мин)
Уточняй ДО рисования архитектуры. Плохие кандидаты сразу рисуют boxes.

- **Use case**: кто пользователь, что он делает, как часто
- **Inputs/outputs**: формат, длина, язык, latency budget
- **Quality bar**: точность, галлюцинации, стоимость ошибки
- **Constraints**: latency (p50/p99), throughput, cost per request, on-prem vs cloud, языки, compliance

Конкретные вопросы которые стоит задать:
- Real-time или batch?
- Single-turn или multi-turn (агент)?
- Streaming нужен?
- Мультимодальный вход (текст/аудио/изображения)?
- Какая tolerance к галлюцинациям?
- Где живут данные (on-prem, AWS, гибрид)?
- Размер команды, кто будет поддерживать?

### 2. Non-functional requirements
- Latency: p50/p95/p99 (для LLM обычно целимся в p95 < 2-5 сек end-to-end)
- Throughput: RPS в пике
- Availability: 99.9%?
- Cost: $/1k requests
- Security: PII, prompt injection, jailbreaks
- Observability: логи, трейсы, метрики
- Reproducibility: версионирование промптов, моделей, датасетов

### 4. Deep dive компоненты (выбери 2-3)
Обычно самые интересные:
- **Retrieval pipeline** — chunking, embedding, reranking, hybrid search
- **LLM serving** — vLLM/TGI/Triton, batching, KV cache, quantization
- **Agent loop** — planning, tool use, error recovery, max iterations
- **Evaluation** — offline eval (golden set, LLM-as-judge) + online (A/B, feedback)
- **Data pipeline** — ingestion, dedup, PII removal, labeling
- **Observability** — Langfuse/LangSmith/Phoenix + custom metrics

### 5. Trade-offs
Всегда озвучивай:
- Latency ↔ Quality (больше токенов контекста → лучше ответ, медленнее)
- Cost ↔ Quality (большая модель дороже, точнее)
- Build ↔ Buy (fine-tune vs prompt vs RAG vs API провайдер)
- Freshness ↔ Cost (re-index vector store каждую минуту vs час)
- Throughput ↔ Latency (batching увеличивает throughput, бьёт по latency)

## Типовые задачи, которые дают на такие роли

Учитывая специфику вакансии (Plata — финтех, customer care, compliance, telesales, OCR/ASR), готовься к:

### A) RAG-система для internal knowledge base
- Customer support agent по документации Plata
- Compliance: проверка ответов на соответствие регуляторике
- Продакшен в AWS, Go/Python

### B) Content moderation / abuse detection на LLM
- Классификация контента (toxicity, fraud, spam)
- Multi-label, multi-language
- Trade-off recall vs precision, false positives дороже false negatives (или наоборот)

### C) OCR/ASR pipeline с LLM post-processing
- Voice/OCR сервис → текст → LLM correction/extraction/classification
- Может быть агент с tool use

### D) AI Gateway / multi-model routing
- Единая точка входа для всех LLM-вызовов в компании
- Rate limiting, cost tracking, fallbacks, A/B routing
- Это очень契合 с тем, что они делают ("AI gateway" прямо в описании команды)

### E) Agentic workflow для telesales/customer care
- Copilot для оператора, multi-step, tool use
- Human-in-the-loop, audit trail

## Failure modes, которые обязательно обсуди

Интервьюер из MLOps точно спросит "what breaks in production":

1. **Prompt injection** — враждебный input обходит system prompt
   - Mitigation: input validation, separate contexts, output filtering
2. **Hallucinations** — модель выдумывает
   - RAG, constrained decoding, fact-checking, citations
3. **Distribution shift** — данные в проде отличаются от train
   - Monitoring, drift detection, periodic re-eval
4. **Cost explosion** — пользователь заставил агента вызвать 1000 tools
   - Max iterations, cost limits, timeouts
5. **PII leak** — модель выдала персональные данные
   - PII detection на входе и выходе, redaction, audit
6. **Latency spike** — model server перегружен
   - Auto-scaling, queue management, fallbacks на smaller model
7. **Vendor lock-in** — привязка к OpenAI API
   - Abstraction layer, model-agnostic interface, можно swap
8. **Reproducibility** — нельзя воспроизвести ответ
   - Pin model version, temperature=0, log everything

## Метрики, которые стоит упомянуть

**Quality:**
- Accuracy/F1 на golden set
- LLM-judge score (с ground truth каждые N примеров)
- Human eval (sampled)
- Win rate vs baseline
- Citation precision/recall (для RAG)

**System:**
- p50/p95/p99 latency
- Tokens/sec (throughput)
- Cost per request, cost per 1k tokens
- Error rate, timeout rate
- Cache hit rate

**Business:**
- CSAT, thumbs up/down
- Task completion rate
- Escalation rate to human
- Time saved

## Частые ошибки кандидатов

- ❌ Сразу прыгать в решение, не выяснив требования
- ❌ Рисовать красивую диаграмму без обоснования
- ❌ Игнорировать cost/latency (MLOps интервьюер это любит)
- ❌ Не упомянуть observability и eval
- ❌ "Use Kubernetes" без объяснения зачем
- ❌ Не обсуждать failure modes
- ❌ Фикситься на одной технологии ("всё на LangChain")

## Что повторить/потренировать за день до

1. **RAG deep dive** — chunking strategies, hybrid search, reranking, evaluation. Это 80% LLM system design.
2. **LLM serving** — vLLM vs TGI vs Triton, batching, quantization (INT8, INT4, GPTQ, AWQ, GGUF)
3. **Agent design** — ReAct, tool use, planning, memory, error recovery
4. **Eval methodology** — offline vs online, golden sets, LLM-as-judge
5. **Prompt engineering** — few-shot, CoT, structured outputs, function calling
6. **Vector search** — HNSW, IVF, recall@K, embedding model choice
7. **Cost math** — посчитай $/1M tokens для типичных моделей, посчитай cost per request для RAG с 5k контекста
